# Django Admin Flows -- операции сопровождения NeoMarket

Описание действий, которые выполняются через Django Admin каждого из трёх сервисов. Django Admin -- инструмент для сопровождающих сервиса, не для конечных пользователей (продавцов и покупателей).

> **Соглашения**: все ID -- UUID (string, format: uuid). Цены -- integer в копейках. Все поля -- snake_case. Подробнее: `type-unification.md`.

---

## Кто такие «сопровождающие сервиса»

Каждый сервис NeoMarket -- отдельное Django-приложение со своей БД и своей Django Admin.

| Сервис | Роль | Кто в реальности |
|--------|------|------------------|
| B2B | Оператор склада | Принимает накладные, управляет остатками |
| B2B | Администратор справочников | Управляет категориями, характеристиками (seed-данные) |
| B2B | Support-инженер | Аварийное управление товарами и SKU |
| Moderation | Тимлид модерации | Очередь модерации, разблокировка, статистика |
| Moderation | Администратор справочников | Причины блокировки |
| B2C | Оператор заказов | Смена статусов заказов (сборка, доставка) |
| B2C | Контент-менеджер | Подборки, баннеры |
| B2C | Оператор поддержки | Просмотр корзин, аварийная очистка |

**Зачем Django Admin**: действия, которые нельзя или не нужно делать через API. Приёмка накладных, ручная смена статусов заказов, управление seed-данными, аварийные операции.

## B2B Admin

### ADM-B2B-1: Приёмка накладных

Оператор склада принимает поставку товара от продавца. Приёмка может быть полной или частичной.

#### Что видит оператор

- Список накладных с фильтром по статусу (PENDING / ACCEPTED / PARTIALLY_ACCEPTED / REJECTED)
- При открытии накладной -- таблица позиций: SKU, название, заявленное количество (`quantity`), поле для ввода принятого количества (`accepted_quantity`)

#### Действия

1. Открыть накладную в статусе `PENDING`
2. Для каждой позиции указать `accepted_quantity` (0 <= accepted_quantity <= quantity)
3. Нажать действие «Принять накладную»

#### Побочные эффекты

Для каждой позиции накладной:

```
SKU.active_quantity += accepted_quantity
```

Обновление остатков и статуса накладной -- **в одной транзакции**.

#### Определение статуса накладной

| Условие | Статус |
|---------|--------|
| Все `accepted_quantity == quantity` | `ACCEPTED` |
| Хотя бы один `accepted_quantity > 0`, но не все полностью | `PARTIALLY_ACCEPTED` |
| Все `accepted_quantity == 0` | `REJECTED` |

#### Валидация

- Нельзя принять накладную, которая не в статусе `PENDING`
- `accepted_quantity` не может быть отрицательным
- `accepted_quantity` не может быть больше `quantity`

#### Реализация

Под капотом вызывается тот же endpoint `POST /api/v1/invoices/{id}/accept` (см. [B2B-6](b2b-flows.md#b2b-6-создание-и-приёмка-накладной)). Можно реализовать как Django Admin action, который собирает данные из inline-формы и вызывает эту логику.

### ADM-B2B-2: Управление категориями (seed)

Категории товаров -- справочные данные, загружаемые при развёртывании. Django Admin позволяет управлять ими вручную.

#### Модель

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | PK, `gen_random_uuid()` |
| name | string | Название категории |
| path | ltree | Путь в дереве (`electronics.phones.ios`) |

#### Что видит администратор

- Дерево категорий (или плоский список с отображением path)
- Фильтр по родительскому узлу

#### Действия

- Создать категорию (указать name и родителя)
- Переместить категорию (изменить path)
- Деактивировать категорию

#### Ограничения

- Нельзя удалять категорию, к которой привязаны товары. Сначала нужно перенести товары в другую категорию или деактивировать их.

#### Загрузка данных

Начальные данные загружаются из fixtures (`python manage.py loaddata categories.json`) или management command с CSV-файлом.

### ADM-B2B-3: Управление характеристиками (seed)

Характеристики и их значения -- справочные данные.

#### Модели

**Characteristic** (характеристика):

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | PK |
| name | string | Название: Бренд, Цвет, Размер, Объём памяти и т.д. |

**CharacteristicValue** (значение характеристики):

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | PK |
| characteristic_id | UUID | FK -> Characteristic |
| value | string | Значение: Apple, Samsung, Красный, 256 ГБ и т.д. |

#### Что видит администратор

- Список характеристик с inline-списком значений
- Фильтр по названию характеристики

#### Действия

- CRUD характеристик
- CRUD значений характеристик (inline к характеристике)
- Опционально: привязка характеристик к категориям (какие характеристики доступны для какой категории)

#### Загрузка данных

Начальные данные -- из fixtures или management command.

### ADM-B2B-4: Аварийное управление товарами

Операции support-инженера. Это не штатный рабочий процесс -- используется для отладки, тестирования и разрешения проблемных ситуаций.

#### Что видит support

- Список всех товаров с фильтрами:
  - По статусу: `CREATED`, `ON_MODERATION`, `MODERATED`, `BLOCKED`, `HARD_BLOCKED`
  - По продавцу (`seller_id`)
  - По категории
  - По флагу `deleted`
- Карточка товара: все поля, список SKU, текущий статус, `blocking_reason`, `field_reports`

#### Действия

- Просмотр полной информации о товаре (включая поля, недоступные через API: внутренние метки)
- Ручное изменение статуса (например, `force-MODERATED` для тестирования интеграции с B2C)
- Просмотр `blocking_reason` и `field_reports` (данные, пришедшие от Moderation)

#### Пометка

Все действия через admin-панель логируются. Ручное изменение статуса -- аварийная операция, не замена штатного процесса модерации.

---

### ADM-B2B-5: Просмотр SKU и остатков

#### Что видит оператор / support

- Таблица SKU: товар, название, цена (в копейках), `active_quantity`, `reserved_quantity`
- Фильтры:
  - Нулевые остатки (`active_quantity = 0`)
  - Есть резервы (`reserved_quantity > 0`)
  - По товару / продавцу

#### Действия

- Просмотр детальной информации о SKU
- Ручная корректировка `active_quantity` (аварийно, при расхождении с физическим складом)

#### Ограничения

- Ручная корректировка `reserved_quantity` запрещена -- резервы управляются B2C через `reserve`/`unreserve`/`fulfill`
- При ручном изменении `active_quantity` -- обязательный комментарий в логе (причина корректировки)

## Moderation Admin

### ADM-MOD-1: Справочник причин блокировки

CRUD по таблице `product_blocking_reasons`. Начальные данные -- seed из 10 записей (см. [MOD-6](moderation-flows.md#flow-mod-6-справочник-причин-блокировки)).

#### Модель

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | PK, `gen_random_uuid()` |
| title | string | Текст причины |
| hard_block | boolean | `true` = перманентная блокировка, `false` = мягкая |

#### Что видит администратор

- Список причин блокировки
- Фильтр по `hard_block`

#### Действия

- Создать причину (title + hard_block)
- Редактировать текст причины
- Деактивировать (если причина больше не актуальна)

#### Seed-данные (10 записей)

| title | hard_block |
|-------|:---:|
| Описание не соответствует товару | false |
| Изображение не соответствует товару | false |
| Некорректная категория товара | false |
| Недостаточно информации о товаре | false |
| Нецензурные или оскорбительные материалы | false |
| Дублирование существующего товара | false |
| Некорректная цена | false |
| Контрафактный товар | true |
| Товар запрещён к продаже на территории РФ | true |
| Товар нарушает авторские права | true |

### ADM-MOD-2: Просмотр очереди модерации

Тимлид модерации видит состояние очереди без необходимости «брать» карточки через get-next.

#### Что видит тимлид

- Таблица `product_moderation` с фильтрами:
  - По статусу: `PENDING`, `IN_REVIEW`, `MODERATED`, `BLOCKED`, `HARD_BLOCKED`
  - По очереди: `queue_priority` (1-4)
  - По модератору: `moderator_id`
  - По дате: `date_created`, `date_updated`
- Детали записи:
  - `json_before` / `json_after` (снимки данных товара из B2B)
  - `blocking_reason`, `moderator_comment`
  - Список `field_reports` (замечания по полям)

#### Действия

- Только просмотр. Тимлид не принимает решение по карточке через admin -- для этого есть API модерации (get-next, approve, decline).

---

### ADM-MOD-3: Ручная разблокировка и переназначение

Тимлид модерации разрешает нештатные ситуации.

#### Действия

**Вернуть в очередь (BLOCKED -> PENDING):**
- Тимлид может вернуть карточку в очередь, если блокировка была ошибочной
- `status` -> `PENDING`, `moderator_id` -> null
- Карточка попадает в очередь 2 (исправленные после блокировки)
- Событие в B2B **не отправляется** -- карточка просто возвращается в очередь модерации

**Переназначить модератора:**
- Сменить `moderator_id` на другого модератора (при болезни, отпуске, ошибке)
- Статус остаётся `IN_REVIEW`

**Снять HARD_BLOCKED (только суперадмин):**

> **Аварийное действие.** Снятие HARD_BLOCKED -> PENDING доступно ТОЛЬКО суперадмину. Требования:
> - Mandatory field: `reason` (причина отзыва, min 20 символов)
> - Audit log запись с before/after значениями
> - Уведомление продавцу (email / через dashboard)
> - Это НЕ штатный процесс, а data-fix для ошибок модерации

- Аварийная операция, требует прав суперпользователя Django
- `status` -> `PENDING`, `moderator_id` -> null, `blocking_reason_id` -> null
- Карточка попадает в очередь 1 (как новая)
- Обязательный комментарий с причиной снятия блокировки

---

### ADM-MOD-4: Статистика модераторов

Тимлид модерации отслеживает продуктивность команды.

#### Что видит тимлид

- Количество обработанных карточек по каждому модератору за выбранный период
- Среднее время обработки (от `IN_REVIEW` до решения, т.е. разница `date_moderation - date_updated` для карточек, где `moderator_id` совпадает)
- Соотношение решений: одобрено (MODERATED) / отклонено (BLOCKED) / жёсткая блокировка (HARD_BLOCKED)

#### Реализация

Можно реализовать как:
- Отдельная страница в Django Admin (custom view / `ModelAdmin` с `changelist_view`)
- Или фильтры + агрегации на стандартном списке `product_moderation`

## B2C Admin

### ADM-B2C-1: Управление статусами заказов

Оператор проводит заказ по жизненному циклу: от оплаты до доставки. В учебном проекте нет WMS и курьерской системы, поэтому смена статусов -- ручная.

#### State machine (напоминание)

```
PAID -> ASSEMBLING -> DELIVERING -> DELIVERED
```

Допустимые переходы -- только вперёд по цепочке. Перескакивать и возвращать нельзя.

#### Что видит оператор

- Список заказов с фильтрами:
  - По статусу: `PAID`, `ASSEMBLING`, `DELIVERING`, `DELIVERED`, `CANCELLED`, `CANCEL_PENDING`
  - По дате создания
  - По `user_id`
- Детали заказа:
  - Позиции (`order_items`): `product_title`, `sku_name`, `quantity`, `unit_price`, `line_total`
  - `total_amount` (общая сумма в копейках)
  - `delivery_address`
  - Даты: `created_at`, `updated_at`

#### Действия

**Смена статуса (actions или кнопки):**

| Действие | Переход | Побочный эффект |
|----------|---------|-----------------|
| «Начать сборку» | PAID -> ASSEMBLING | Нет |
| «Отправить» | ASSEMBLING -> DELIVERING | Нет |
| «Доставлен» | DELIVERING -> DELIVERED | POST /api/v1/fulfill к B2B |
| «Отменить заказ» | PAID / ASSEMBLING -> CANCELLED | POST /api/v1/unreserve к B2B |

**Побочный эффект при DELIVERED:**

При переходе в `DELIVERED` B2C вызывает `POST /api/v1/fulfill` к B2B для финального списания резерва:

```
POST {b2b_url}/api/v1/fulfill
X-Service-Key: {b2c_to_b2b_key}

{
  "order_id": "<uuid заказа>",
  "items": [
    {"sku_id": "<uuid>", "quantity": <int>},
    ...
  ]
}
```

`reserved_quantity -= quantity` для каждого SKU. Без этого `reserved_quantity` копится бесконечно (см. [B2C-13](b2c-orders-flows.md#flow-b2c-13-fulfill--списание-резерва-при-delivered)).

**Побочный эффект при отмене оператором:**

При переходе в `CANCELLED` B2C вызывает `POST /api/v1/unreserve` к B2B:

```
POST {b2b_url}/api/v1/unreserve
X-Service-Key: {b2c_to_b2b_key}

{
  "order_id": "<uuid заказа>",
  "items": [
    {"sku_id": "<uuid>", "quantity": <int>},
    ...
  ]
}
```

`active_quantity += quantity`, `reserved_quantity -= quantity` для каждого SKU.

**Реализация побочных эффектов:**
- Django Admin action, который после смены статуса вызывает B2B API
- Или Django signal (`post_save`) на модели Order, который при смене статуса на `DELIVERED` вызывает fulfill, а при `CANCELLED` -- unreserve
- Решение за командой. Оба варианта допустимы.

#### Валидация

- Нельзя перескочить статус (PAID -> DELIVERING -- ошибка)
- Нельзя вернуть статус назад (DELIVERING -> ASSEMBLING -- ошибка)
- Нельзя менять статус заказа в `DELIVERED` или `CANCELLED` (терминальные)
- Нельзя менять статус `CANCEL_PENDING` (управляется асинхронной задачей)

### ADM-B2C-2: Подборки товаров (Collections)

Контент-менеджер формирует подборки товаров для отображения на главной странице B2C.

#### Модель

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | PK |
| title | string | Название подборки: «Хиты продаж», «Новинки» |
| product_ids | UUID[] | Массив ID товаров из B2B |
| is_active | boolean | Отображать на главной |
| ordering | integer | Порядок отображения |

#### Что видит контент-менеджер

- Список подборок с фильтром по `is_active`
- Детали подборки: название, список product_ids, порядок

#### Действия

- Создать подборку: ввести title, список UUID товаров, порядок
- Редактировать подборку: изменить состав товаров, название, порядок
- Активировать / деактивировать подборку

#### Валидация product_ids

B2C хранит только UUID товаров, не копию данных. При отображении подборки на витрине B2C запрашивает данные у B2B через batch-запрос:

```
GET {b2b_url}/api/v1/products?ids=<uuid1>,<uuid2>,...
X-Service-Key: {b2c_to_b2b_key}
```

Товары, которые не прошли условие видимости (не `MODERATED`, `deleted`, нет остатков), просто не попадут в ответ B2B. Подборка покажет только доступные товары.

Валидация product_ids при сохранении в admin **не обязательна** -- достаточно проверки при отображении. Но команда может добавить проверку при сохранении (синхронный запрос к B2B) как дополнительную гарантию.

### ADM-B2C-3: Баннеры

Контент-менеджер управляет баннерами на главной странице B2C.

#### Модель

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | PK |
| title | string | Название баннера (для admin, не отображается покупателю) |
| image_url | string | URL изображения (S3 или локальное хранилище) |
| link_url | string | URL перехода при клике (страница категории, товара, подборки) |
| is_active | boolean | Отображать на главной |
| ordering | integer | Порядок отображения |

#### Что видит контент-менеджер

- Список баннеров с фильтром по `is_active`
- Превью изображения (если возможно)

#### Действия

- Создать баннер: загрузить изображение, указать link_url, порядок
- Редактировать баннер
- Активировать / деактивировать

#### Загрузка изображений

Изображения баннеров хранятся в S3 (или локально в `MEDIA_ROOT`). Загрузка через стандартный Django `FileField` / `ImageField`.

### ADM-B2C-4: Просмотр корзин и избранного

Оператор поддержки помогает покупателям с проблемами в корзине.

#### Что видит оператор

- Поиск корзины по `user_id`
- Содержимое корзины: `sku_id`, `quantity` для каждого элемента
- Содержимое избранного: список `product_id`

#### Действия

- Просмотр корзины конкретного пользователя (только чтение в штатном режиме)
- Ручная очистка корзины (если зависла -- например, после ошибки при checkout)
- Удаление отдельных элементов из корзины

#### Ограничения

- Нельзя добавлять товары в корзину за покупателя
- Нельзя создавать заказы за покупателя

## Общие паттерны для всех Admin

### 1. Аутентификация и авторизация

Стандартный Django auth: `User` + `Group` + `Permission`. Суперпользователь имеет доступ ко всему.

### 2. Группы прав

| Сервис | Группа | Доступ |
|--------|--------|--------|
| B2B | `warehouse_operator` | Накладные: просмотр и приёмка (ADM-B2B-1) |
| B2B | `seed_admin` | Категории (ADM-B2B-2), характеристики (ADM-B2B-3) |
| B2B | `product_support` | Аварийное управление товарами (ADM-B2B-4), SKU и остатки (ADM-B2B-5) |
| Moderation | `moderation_admin` | Справочник причин (ADM-MOD-1) |
| Moderation | `moderator_lead` | Очередь модерации (ADM-MOD-2), разблокировка и переназначение (ADM-MOD-3), статистика (ADM-MOD-4) |
| B2C | `order_operator` | Статусы заказов (ADM-B2C-1) |
| B2C | `content_manager` | Подборки (ADM-B2C-2), баннеры (ADM-B2C-3) |
| B2C | `customer_support` | Просмотр корзин и избранного (ADM-B2C-4) |

### 3. Audit log

Все действия через Django Admin логируются через стандартный `django.contrib.admin.models.LogEntry`. Django записывает: кто, когда, какой объект, какое действие (add/change/delete).

Для операций с побочными эффектами (приёмка накладных, смена статуса заказа) рекомендуется дополнительное логирование: какой межсервисный вызов был сделан, результат, время.

### 4. Побочные эффекты (межсервисные вызовы)

Некоторые admin-действия триггерят вызовы к другим сервисам:

| Действие в Admin | Сервис | Вызов | Описание |
|------------------|--------|-------|----------|
| Приёмка накладной (ADM-B2B-1) | B2B | Внутренний | `active_quantity += accepted_quantity` (в той же БД) |
| Заказ -> DELIVERED (ADM-B2C-1) | B2C -> B2B | `POST /api/v1/fulfill` | Списание резерва |
| Заказ -> CANCELLED (ADM-B2C-1) | B2C -> B2B | `POST /api/v1/unreserve` | Возврат остатков |

Если межсервисный вызов упал -- admin должен показать ошибку оператору. Заказ остаётся в текущем статусе, оператор повторяет действие позже. Для `DELIVERED` при ошибке fulfill: допустимо перевести заказ в DELIVERED и поставить fulfill на async retry (товар уже у покупателя, откатывать статус не имеет смысла).
