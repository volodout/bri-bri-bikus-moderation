# NeoMarket User Stories

Продуктовые user stories для всех 49 flows маркетплейса NeoMarket. Каждая story связывает техническое поведение системы с пользой для конкретного пользователя. Используется командами разработки для понимания смысла фичи до погружения в JSON schemas и sequence diagrams.

> Технические детали (endpoints, request/response): [b2b-flows.md](b2b-flows.md), [moderation-flows.md](moderation-flows.md), [b2c-catalog-flows.md](b2c-catalog-flows.md), [b2c-cart-flows.md](b2c-cart-flows.md), [b2c-orders-flows.md](b2c-orders-flows.md), [admin-flows.md](admin-flows.md), [events-schema.md](events-schema.md).
> Соглашения по типам: [type-unification.md](type-unification.md).

## Формат user story

```
### US-XXX: [краткое название]

**Как** [роль]
**Я хочу** [цель]
**Чтобы** [выгода]

**Acceptance Criteria** (Given-When-Then):
- **Given** [начальное состояние]
- **When** [действие]
- **Then** [результат]

**Ссылки**: flow / endpoint
**Приоритет**: Must / Should / Could
```

## Персоны

### Внешние пользователи

**Покупатель (buyer)**
Физлицо, ищет нужный товар, сравнивает цены, оформляет заказ, отслеживает доставку. Приходит через поиск, каталог или подборки. Мотивация: купить быстро, не переплатить, не столкнуться с недоступным товаром после оплаты.

**Продавец (seller)**
Представитель магазина, добавляет товары, поддерживает остатки, исправляет замечания модератора. Работает через B2B-кабинет. Мотивация: провести товар через модерацию, получать заказы, не терять деньги на зависших резервах.

### Внутренние сотрудники

**Модератор (moderator)**
Проверяет карточки товаров: соответствие описания, фото, цены, категории. Работает с очередью через веб-интерфейс модерации. Мотивация: быстро принимать решения, не пропускать проблемные товары.

**Тимлид модерации (moderation_lead)**
Управляет работой модераторов, справочником причин блокировки, разбирает нештатные ситуации (переназначение, спорные решения). Мотивация: держать очередь короткой, команду продуктивной, минимизировать ошибочные блокировки.

**Оператор склада (warehouse_operator)**
Принимает накладные от продавцов, вводит фактически принятое количество. Меняет статусы заказов (сборка, отгрузка). Мотивация: точно учесть поступление, провести заказ до доставки.

**Контент-менеджер (content_manager)**
Формирует подборки товаров, баннеры для главной страницы. Мотивация: увеличить вовлечённость покупателей, подсветить акции.

**Оператор поддержки (support)**
Решает проблемы покупателей и продавцов. Имеет расширенные права: аварийная смена статусов, очистка корзин. Мотивация: закрывать тикеты, не ломая данные.

**Seed-admin**
Управляет справочниками: категории (ltree-дерево), характеристики и их значения, причины блокировки. Мотивация: держать таксономию согласованной.

**Системный интегратор (service)**
Не человек, а межсервисный вызов. Отдельные stories описывают инварианты асинхронной коммуникации между B2B, Moderation и B2C: идемпотентность, outbox, retry. Мотивация: данные синхронизированы, события не теряются.

## B2B — Кабинет продавца

### US-B2B-01: Создание карточки товара

**Как** продавец
**Я хочу** создать карточку товара с названием, описанием, фото, категорией и характеристиками
**Чтобы** начать процесс подготовки товара к продаже

**Acceptance Criteria**:
- **Given** я авторизован в B2B-кабинете
- **When** заполняю форму создания товара (title, description, category, images, характеристики) и сохраняю
- **Then** товар создаётся со статусом CREATED и без SKU
- **And** я перенаправляюсь на шаг добавления SKU
- **And** товар НЕ отправляется на модерацию (нужен хотя бы один SKU)
- **And** если нет изображений или категории — получаю 400 с описанием, что исправить

**Ссылки**: [B2B-1](b2b-flows.md#b2b-1-создание-товара), `POST /api/v1/products`
**Приоритет**: Must

### US-B2B-02: Добавление варианта товара (SKU)

**Как** продавец
**Я хочу** добавить конкретный вариант товара с ценой, себестоимостью, фото и характеристиками варианта
**Чтобы** выставить товар на продажу и отправить его на модерацию

**Acceptance Criteria**:
- **Given** у меня есть товар в статусе CREATED без SKU
- **When** создаю первый SKU (price, cost_price, image, цвет, размер и т.п.)
- **Then** SKU создаётся, статус товара меняется на ON_MODERATION
- **And** событие CREATED уходит в Moderation
- **And** если я добавляю второй SKU к этому же товару — он просто появляется, статус не меняется, события не отправляются

### US-B2B-08: Резервирование и снятие резерва SKU

**Как** системный интегратор (B2C при checkout и cancel)

**Чтобы** два покупателя не купили один и тот же последний остаток

**Acceptance Criteria**:
- **Given** SKU имеет active_quantity=N и reserved_quantity=M
- **When** B2C вызывает reserve с quantity=K (K <= N)

- **And** повторный запрос с тем же idempotency_key возвращает тот же результат без повторного резервирования
- **And** unreserve делает обратную операцию (active_quantity+=K, reserved_quantity-=K)
- **And** если active_quantity SKU стал 0 после reserve — уходит событие SKU_OUT_OF_STOCK в B2C

**Ссылки**: [B2B-8](b2b-flows.md#b2b-8-reserve--unreserve), `POST /api/v1/reserve`, `POST /api/v1/unreserve`
**Приоритет**: Must

## Moderation — Очередь модерации

### US-MOD-01: Приём событий о товаре от B2B

**Как** системный интегратор (Moderation принимает события от B2B)
**Я хочу** обрабатывать события CREATED / EDITED / DELETED и поддерживать актуальный json_after

- **When** Moderation принимает его с валидным X-Service-Key

- **And** создаётся запись product_moderation со статусом PENDING и queue_priority=1
- **And** при EDITED: json_before=старый json_after, json_after=новый из B2B, статус→PENDING, очередь вычисляется по прошлому статусу и наличию остатков
- **And** при DELETED: запись удаляется (идемпотентно, повторный DELETED возвращает 200)
- **And** для HARD_BLOCKED товаров события EDITED игнорируются

## B2C Admin

### US-ADM-B2C-01: Смена статусов заказов через Django Admin

**Как** оператор склада
**Я хочу** проводить заказ по цепочке PAID → ASSEMBLING → DELIVERING → DELIVERED через Django Admin
**Чтобы** в учебном проекте без реального WMS и курьерки модель заказа двигалась вручную

**Acceptance Criteria**:
- **Given** заказ в PAID
- **When** нажимаю «Начать сборку»
- **Then** статус меняется на ASSEMBLING
- **And** аналогично «Отправить» переводит в DELIVERING, «Доставлен» — в DELIVERED
- **And** при DELIVERED B2C вызывает fulfill в B2B
- **And** при отмене оператором (PAID / ASSEMBLING → CANCELLED) — B2C вызывает unreserve
- **And** нельзя перескакивать статусы (PAID → DELIVERING отклоняется)
- **And** нельзя возвращать назад (DELIVERING → ASSEMBLING отклоняется)
- **And** терминальные статусы (DELIVERED, CANCELLED) менять нельзя
- **And** если межсервисный вызов упал — оператор видит ошибку; для fulfill допустимо DELIVERED + async retry (товар уже у покупателя)

**Ссылки**: [ADM-B2C-1](admin-flows.md#adm-b2c-1-управление-статусами-заказов)
**Приоритет**: Must

## Межсервисные события

### US-EVT-01: События B2B → Moderation (CREATED / EDITED / DELETED)

**Как** системный интегратор
**Я хочу** надёжно доставлять события о жизненном цикле товара из B2B в Moderation с защитой от потери и дублей
**Чтобы** очередь модерации всегда отражала актуальное состояние товаров продавцов

**Acceptance Criteria**:
- **Given** продавец выполнил действие (создал первый SKU, изменил товар, удалил)
- **When** B2B записывает событие в outbox в одной транзакции с изменением данных
- **Then** worker шлёт POST {mod}/api/v1/events/product с idempotency_key, product_id, seller_id, event, date и X-Service-Key
- **And** при 5xx или таймауте — ретраи с экспоненциальным backoff (30с→60с→120с→...) до 10 попыток, затем FAILED и алерт
- **And** при 400 (невалидные данные) — не ретраить, пометить FAILED
- **And** Moderation хранит обработанные idempotency_key и отвечает 200 на повторы без повторного применения
- **And** без валидного X-Service-Key — 401

**Ссылки**: [EVT-1](events-schema.md#1-b2b---moderation), `POST {mod}/api/v1/events/product`
**Приоритет**: Must

### US-EVT-02: События Moderation → B2B (MODERATED / BLOCKED)

**Как** системный интегратор
**Я хочу** доставить решение модератора обратно в B2B с полной информацией о причине и замечаниях
**Чтобы** продавец видел корректный статус и мог исправить замечания, а каталог отражал результат модерации

**Acceptance Criteria**:
- **Given** модератор принял решение (approve или decline)
- **When** Moderation шлёт POST {b2b}/api/v1/events/moderation с X-Service-Key
- **Then** для MODERATED передаются idempotency_key, product_id, status
- **And** для BLOCKED дополнительно передаются hard_block (true/false), blocking_reason (id/title/comment) и field_reports
- **And** B2B обновляет статус, сохраняет замечания и каскадно шлёт PRODUCT_BLOCKED в B2C при блокировке
- **And** повторные события с тем же idempotency_key не применяются дважды
- **And** при 5xx — ретраи, при 400 — не ретраить

**Ссылки**: [EVT-2](events-schema.md#2-moderation---b2b), `POST {b2b}/api/v1/events/moderation`
**Приоритет**: Must

### US-EVT-03: События B2B → B2C (PRODUCT_BLOCKED / PRODUCT_DELETED / SKU_OUT_OF_STOCK)

**Как** системный интегратор
**Я хочу** уведомлять B2C об изменениях доступности товаров
**Чтобы** корзины и избранное у покупателей отражали реальное состояние, а заказы с зафиксированными ценами не трогались

**Acceptance Criteria**:
- **Given** в B2B произошло событие (блокировка модератором, удаление продавцом, остаток SKU стал 0)
- **When** B2B пишет событие в outbox и worker шлёт POST {b2c}/api/v1/events/product с X-Service-Key
- **Then** B2C помечает cart_items и wishlist как unavailable, заказы не трогает
- **And** идемпотентность через idempotency_key — повтор обрабатывается без эффекта
- **And** PRODUCT_BLOCKED и PRODUCT_DELETED содержат product_id и массив sku_ids
- **And** SKU_OUT_OF_STOCK содержит один sku_id и product_id
- **And** при 5xx — retry с backoff, при 400 — не ретраить

**Ссылки**: [EVT-3](events-schema.md#3-b2b---b2c), `POST {b2c}/api/v1/events/product`
**Приоритет**: Must
