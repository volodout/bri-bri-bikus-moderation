# neomarket-canon

Это каноничная база маркетплейса NeoMarket. Здесь живут правила, которым обязаны следовать все сервисы.

---

## Что тут есть

### flows/ — это ЗАКОН

Папка `flows/` содержит каноничные user flows: как ведёт себя система при каждом пользовательском действии. Продавец создаёт товар, покупатель оформляет заказ, модератор принимает решение — всё это здесь.

Если твой сервис реализует какой-то из этих flows, он **обязан** ему следовать. При сдаче работы peer-reviewer проверит соответствие canonic flow.

Начни с [`flows/INDEX.md`](flows/INDEX.md) — там навигация и полная карта.

### apis/ — это РЕФЕРЕНС (опционально)

Папка `apis/` будет содержать эталонные OpenAPI-спецификации. Они не обязательны — ты можешь писать свои. Но если есть готовый эталон, имеет смысл на него ориентироваться.

В текущем семестре `apis/` пустая: спецификации создаёте вы сами в процессе разработки.

---

## Как пользоваться

1. **Открой нужный flow** в `flows/`. Например, если делаешь B2B кабинет продавца — смотри [`flows/b2b-flows.md`](flows/b2b-flows.md).
2. **Разберись в flow** — шаги пользователя, endpoint, request/response schemas, edge cases.
3. **Реализуй сервис** так, чтобы он проходил этот flow.
4. **Опиши OpenAPI** своего сервиса в репозитории neomarket-protocols.

---

## Как описать OpenAPI (и получить бонус)

Когда ты реализовал сервис и хочешь поделиться спецификацией с другими командами — сделай Pull Request в репозиторий **neomarket-protocols**.

Там действует механизм, по которому автор PR и все, кто оставил содержательный review-комментарий, получают бонус. Это выгодно: другие команды смогут сверяться с твоей спецификацией, а не изобретать велосипед.

Подробности о процессе — в `neomarket-protocols/docs/`.

---

## Вопросы

Нашёл несоответствие между canon flow и реальностью? Или flow непонятен? Открой issue **в репозитории neomarket-protocols** — там живёт координация между командами.

Вопросы по содержимому конкретного flow можно задавать здесь через issues тоже.

## CI

В репозитории настроены три проверки:

| Workflow | Что проверяет | Режим |
|----------|---------------|-------|
| `pii-guard` | Имена и TG-хэндлы в текстах | warning-only |
| `link-validator` | Внутренние ссылки между файлами | warning-only |
| `schema-syntax` | YAML front-matter + mermaid блоки | warning-only |

Все три — warning-only: они не блокируют PR, но сигнализируют о проблемах.
